В проекте создал обложку (Banking), под которую спрятал весь функционал. Внутри есть Коннектор к БД, менеджеры по работе с юзерами, счетами, транзакциями (по одному на каждую сущность). Обработка транзакций выполняется процессором. 

## Ответы на вопросы

### 1. Что такое UUID?
UUID — это универсальный уникальный идентификатор. Обычно такого вида: xxxxxxxx-xxxx-Mxxx-Nxxx-xxxxxxxxxxxx -- 36 символов. Генерируется случайным образом. Вероятность того, что два UUID совпадут, практически нулевая. 

### 2. Архитектура: "Индекс списка как ID"
**Вопрос:** Нормально ли в транзакции указывать номер счета в списке (индекс)?
**Ответ:** **Нет, это плохая практика.** Списки меняются: если удалить первый счет, второй станет первым. Изменится индексация -- всё сломается.
**Решение:** хранить UUID счета.

### 3. Принципы SOLID
*   **S:** Разделил логику:  работает с БД,  с логикой, с интерфейсом.
*   **O:** Новые операции добавляются через новые классы команд (паттерн Command), не меняя старый код.
*   **L/I/D:** Везде используются интерфейсы. Нет жестких зависимостей.

### 4. Паттерны проектирования
*   **Command:** Каждой транзакции ставится в соответствие команда, которая потом выполняется на процессоре транзакций.
*   **Factory:** Фабрика создает нужную транзакцию.
*   **Singleton:**  Используется для создания приложения (app) и DataBaseConnection.

### 5. Асинхронность
*   Используются блокировки, синхронизация, volatile, Atomic, все важные участки обрабатываются лишь одним потоком.
*   Создается объект ExecutorService, который создает фиксированное кол-во потоков, на который нацепляются процессы.
*   Состояние гонки не достигается.
